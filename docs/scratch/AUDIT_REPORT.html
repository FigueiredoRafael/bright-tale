🛡️ Segurança e Banco de Dados

Abuso de Service Role em Políticas de RLS

LOCAL: supabase/migrations/*_affiliate_*.sql
Crítico

A maioria das tabelas de afiliados e logs de motor está utilizando a política service_role_all. Isso remove a segunda camada de defesa do banco de dados, confiando 100% que a API nunca será comprometida.

Exemplo de Vulnerabilidade (SQL)
-- Falha: Sem isolamento de usuário CREATE POLICY "service_role_all" ON public.affiliates FOR ALL TO service_role USING (true); -- Sugestão: Defesa em Profundidade CREATE POLICY "users_read_own_data" ON public.affiliates FOR SELECT TO authenticated USING (auth.uid() = user_id);

💳 Consumo de Créditos e Billing NEW

Ausência de Atomicidade e Hold de Créditos

LOCAL: apps/api/src/lib/credits.ts
Crítico

A dedução de créditos ocorre de forma assíncrona após a execução da tarefa. Não há transações SQL para garantir que o saldo não fique negativo sob carga simultânea (Race Condition).

Risco de Race Condition (Node.js)
// Passo 1: Verifica saldo (sem lock) const balance = await getBalance(orgId); // Passo 2: Executa tarefa cara (ex: chama IA por 30s) await callExpensiveAI(); // Passo 3: Debita (Saldo pode ter mudado!) await debitCredits(orgId, cost);
Risco de Perda Se o processo cair no Passo 2, os créditos nunca são debitados, gerando prejuízo operacional.
Gasto Duplo Usuários podem disparar múltiplas requisições paralelas e "burlar" o limite de saldo disponível.

🔄 Inconsistências de Motor

Fragmentação do IdeaContext

LOCAL: apps/api/src/lib/ai/loadIdeaContext.ts vs assets.ts
Média

O motor de Assets não consome o IdeaContext estruturado. Enquanto o motor de Draft usa loadIdeaContext para extrair tom e audiência, o motor de assets faz queries ad-hoc ou recebe dados parciais, resultando em imagens menos alinhadas ao contexto original.

Dualidade Legacy/JSONB em Drafts

LOCAL: apps/api/src/routes/content-drafts.ts
Alta

A migração para o campo draft_json está em "estado fantasma". Algumas rotas ainda tentam extrair campos como blog_content diretamente, enquanto o novo padrão armazena tudo no JSONB.

Estratégia de Remediação

Fase 1: Hardening & Billing (24h) Implementar transações SQL no debitCredits e políticas de RLS granulares.
Fase 2: Unificação (48h) Integrar 'loadIdeaContext' no AssetsEngine e padronizar o acesso ao 'draft_json'.
Fase 3: Observabilidade Expandir logs do Axiom para todos os fluxos de faturamento e revisão.
