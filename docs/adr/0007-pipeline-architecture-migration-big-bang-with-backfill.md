# Pipeline architecture migration — big bang with backfill

Existing Projects are migrated to the new orchestration architecture in a single deploy via a one-off backfill script that derives `stage_runs` rows from each Project's `pipeline_state_json` (one `completed` Stage Run per `stageResults` entry, one `queued` or `awaiting_user` Stage Run for the `currentStage`, Mode/Paused derived from legacy fields). Alternatives considered: per-Project feature flag with dual legacy/v2 codepaths kept alive until all legacy Projects finish (rejected — multi-month maintenance burden of two pipelines for a benefit not justified by the small population of in-flight Projects); hard cutover invalidating in-flight work (rejected — rude to the user, and unnecessary given the backfill is deterministic). Backfill is run once at deploy time; Projects that cannot be migrated cleanly are logged and quarantined for manual review rather than silently corrupted.
