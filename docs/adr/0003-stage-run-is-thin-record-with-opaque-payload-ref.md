# Stage Run is a thin record with opaque Payload Ref

A Stage Run row carries only orchestration metadata (`project_id`, `stage`, `status`, `started_at`, `finished_at`, `error_message`, `payload_ref`) — never the domain payload. `payload_ref` is an opaque `{ kind, id }` pointer (jsonb) referencing the existing per-stage payload table (`research_sessions`, `brainstorm_drafts`, `content_drafts`, etc.). Alternatives considered: typed foreign keys per stage (rejected — schema changes every time a new Stage type is added, e.g. shorts/podcast in yt-pipeline-v2); single mega-table with polymorphic `payload_json` (rejected — loses typed payloads, breaks existing mappers, expensive migration). The hybrid keeps orchestration uniform across Stages while leaving payload schemas owned by the stage that produces them. The Pipeline Orchestrator never reads the payload — only the Engine resolves it when the user opens that Stage.
