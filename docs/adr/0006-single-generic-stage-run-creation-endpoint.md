# Single generic Stage Run creation endpoint

Creating a Stage Run goes through one endpoint — `POST /api/projects/:projectId/stage-runs` — for every Stage, with stage-specific input validated by a dispatcher (`{ research: researchInputSchema, brainstorm: brainstormInputSchema, ... }`). The endpoint handles auth, ownership, predecessor-completeness check, UNIQUE-constraint enforcement, and Inngest enqueue uniformly. Existing per-stage endpoints (`POST /api/research-sessions`, etc.) remain as deprecated thin proxies until callers migrate. Alternatives considered: keep per-stage endpoints (rejected — duplicates auth/ownership/validation/logging across 6+ routes and forces a new endpoint for every new Stage type); GraphQL/tRPC (rejected — overkill for the scope; the repo already standardises on the `{data,error}` envelope). The same endpoint is used by both the browser (manual mode) and the Pipeline Orchestrator (autopilot, internally), guaranteeing one canonical creation path with zero behavioural divergence between modes.
