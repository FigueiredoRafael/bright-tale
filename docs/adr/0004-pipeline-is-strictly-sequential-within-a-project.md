# Pipeline is strictly sequential within a Project

A Project advances one Stage at a time — at most one non-terminal Stage Run exists per Project. Concurrency exists only between Projects (each Project advances independently, and the Pipeline Orchestrator handles them in parallel via Inngest). Within a Project, no Stage runs in parallel with any other. Alternatives considered: a DAG model where `assets` and `shorts derivation` (or `podcast generation`) could fan out from `draft` and join back at `preview` — rejected because the product owner has no use case for it and the complexity (multi-predecessor join logic, partial-failure semantics, more error surface in the orchestrator) is not warranted. If parallel branches become needed later, this ADR can be superseded; the change touches `nextStageAfter()` (returns array) and the orchestrator's "ready to advance" guard, but no existing Stage Run data needs to migrate.
