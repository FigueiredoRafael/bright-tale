# Pipeline Orchestrator is event-driven via Inngest

The Pipeline Orchestrator is implemented as an Inngest function (`pipeline-advance`) that reacts to a `stage.run.finished` event. Each `*-generate` job emits this event when its Stage Run reaches a terminal status; `pipeline-advance` consults `projects.mode`, `projects.paused`, and the autopilot config to decide whether to enqueue the next Stage Run. Alternatives considered: each job enqueues its own successor (rejected — duplicates transition logic across 6 files); cron poller (rejected — adds latency and polling noise); Postgres trigger (rejected — hard to test/observe and inconsistent with the rest of the Inngest-based codebase). Event-driven via Inngest gives a single locus for transition logic, native replay/retry, and consistency with how every other long-running operation in the system is already orchestrated.
