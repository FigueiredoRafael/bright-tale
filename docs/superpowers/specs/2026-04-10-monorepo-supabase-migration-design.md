# Monorepo + Supabase Migration — Design Spec

**Date:** 2026-04-10
**Status:** Approved for planning
**Author:** Claude (brainstorm session with Thiago Figueiredo)

## Context

The `bright-tale` repo currently holds a single Next.js app at `bright-curios-workflow/` using Prisma + PostgreSQL. We are restructuring into an npm-workspaces monorepo mirroring the conventions of `~/Workspace/tonagarantia`, and replacing Prisma with Supabase. Everything remains Next.js (no React Native, no Fastify).

**Target layout:**

- `apps/app` → `app.brighttale.io` (main product UI, migrated from `bright-curios-workflow`)
- `apps/api` → `api.brighttale.io` (Next.js dedicated to Route Handlers)
- `apps/web` → `brighttale.io` (landing page, new)
- `packages/shared` → `@brighttale/shared` (types, schemas, mappers)

**Key decisions already taken:**

- **Schema**: port from Prisma to Supabase SQL migrations (Option C); structure preserved, data discarded.
- **Auth**: deferred (Option B). All traffic routes through `apps/api` using `service_role`. Real auth via `@tn-figueiredo/auth-nextjs` in a later spec.
- **Shared package consumption**: source alias + `transpilePackages` (not the `dist/` pattern from tonagarantia — better DX).
- **Reference monorepo**: tonagarantia, mirror all conventions except where noted.

### 1.2 — Local development topology

| App | Port | Local URL |
|---|---|---|
| `apps/app` | 3000 | `http://localhost:3000` |
| `apps/api` | 3001 | `http://localhost:3001` |
| `apps/web` | 3002 | `http://localhost:3002` |

**How `apps/app` talks to `apps/api`:**

- `apps/app/next.config.ts` declares rewrites: `/api/:path*` → `${process.env.API_URL}/api/:path*`
- Browser sees everything same-origin (`localhost:3000/api/projects`). No CORS.
- Next server transparently proxies to `localhost:3001` in dev, `api.brighttale.io` in prod.
- **The `/api/` prefix is preserved** — simplifies migration (just move `src/app/api/**` wholesale, no path renames).

**Dev orchestration:**

- Root script `dev` uses `concurrently` to start `apps/app` + `apps/api` together (normal dev case).
- `dev:web` separate (rarely run together).
- Individual scripts: `dev:app`, `dev:api`, `dev:web`.

### 1.3 — Security in the no-auth phase (Option B)

- `apps/api` is publicly reachable at `api.brighttale.io`. **Risk:** without protection, anyone can query/mutate data.
- Interim protection: **`INTERNAL_API_KEY` shared secret**.
  - `apps/app` server-side (rewrites + `lib/api-client.ts`) injects `X-Internal-Key` header.
  - `apps/api/src/app/middleware.ts` rejects `/api/*` without valid header with 401.
  - Value: `openssl rand -hex 32`, env var in both apps, same key.
- **Not user authentication** — anti-drive-by barrier until real auth lands.
- **RLS enabled on ALL tables from day 1, zero policies = deny-all for `anon`/`authenticated`**. `service_role` bypasses RLS by design. When real auth lands, we add policies — structure doesn't change.
- `NEXT_PUBLIC_SUPABASE_ANON_KEY` stays in `apps/app/.env.example` but is **unused today** — provisioned for the auth switchover.

### 1.4 — Deploy topology (Vercel)

Three separate Vercel projects (same convention as tonagarantia):

| Project | Domain | Root directory |
|---|---|---|
| `@brighttale/web` | `brighttale.io`, `www.brighttale.io` | `apps/web` |
| `@brighttale/app` | `app.brighttale.io` | `apps/app` |
| `@brighttale/api` | `api.brighttale.io` | `apps/api` |

Each app has its own minimal `vercel.json` (Next auto-detects). `packages/shared` does not deploy — it's a library consumed by each build.

### 1.5 — `packages/shared` consumption (deviation from tonagarantia)

**Decision:** use **source alias + `transpilePackages`** instead of tonagarantia's "build dist, consume dist" pattern. Rationale: much better DX (edit shared → hot reload in apps) and Next.js supports it natively.

- `packages/shared/package.json`:
  ```json
  {
    "name": "@brighttale/shared",
    "exports": {
      ".": {
        "types": "./src/index.ts",
        "default": "./src/index.ts"
      }
    }
  }
  ```
- Each `apps/*/next.config.ts`:
  ```ts
  transpilePackages: ['@brighttale/shared']
  ```
- Each `apps/*/tsconfig.json` has paths:
  `"@brighttale/shared": ["../../packages/shared/src"]`,
  `"@brighttale/shared/*": ["../../packages/shared/src/*"]`.
- `build:shared` exists (for future external publishing / CI checks) but is **not a prerequisite** for `build:app` / `build:api` / `build:web`.

### 1.6 — Supabase topology

**Two Supabase projects:** `bright-tale-dev` and `bright-tale-prod`.

> ⚠️ The credentials provided (`mzdtknroizehxrjptlwd`) will be treated as **prod** unless otherwise instructed. **Before running the first migration, a separate `bright-tale-dev` Supabase project must be created.**

Structure in `supabase/`:

- `config.toml` — generated by `supabase init`
- `migrations/` — incremental, numbered
- `seed.sql` — generated by `scripts/generate-seed.ts` (reads `agents/*.md`)

Root scripts (mirror tonagarantia):

- `db:start`, `db:stop`, `db:reset`, `db:status` — local CLI (Docker)
- `db:link:dev`, `db:link:prod` — link CLI to remote project (with visually contrasting echo of the ref to prevent accidents)
- `db:push:dev`, `db:push:prod` — apply migrations; `db:push:prod` goes through `scripts/db-push-prod.sh` which asks for confirmation
- `db:types` — `supabase gen types typescript --local > packages/shared/src/types/database.ts`
- `db:seed` — generates seed.sql + local reset + apply
- `db:which` — shows which project the CLI is linked to (safety)

**Generated types:** `packages/shared/src/types/database.ts` — **committed**. Rebuild every time a migration changes schema.

### 1.7 — Prisma → SQL translation rules

| Prisma | SQL Postgres |
|---|---|
| `String @id @default(cuid())` | `id text primary key default gen_random_uuid()::text` |
| `String` | `text` |
| `String?` | `text` (nullable) |
| `Int` | `integer` |
| `Boolean` | `boolean` |
| `DateTime @default(now())` | `timestamptz not null default now()` |
| `DateTime? @updatedAt` | column + `moddatetime` trigger (extension) |
| `Json` | `jsonb` |
| `enum X { A, B }` | `create type x as enum ('A','B')` — preserves casing |
| `@relation(onDelete: Cascade)` | `on delete cascade` on FK |
| `@@index([field])` | `create index idx_<table>_<field> on ...` |
| `@@unique([a,b])` | `create unique index ... on (a,b)` |
| Self-ref (`Template.parent`) | FK `deferrable initially deferred` |
| camelCase field in Prisma | **snake_case column in SQL** (Postgres convention) |

**Naming:** everything snake_case in the database. Types generated by `supabase gen types` are snake_case. **Mappers in `packages/shared/src/mappers/db.ts`** convert camelCase ↔ snake_case to keep BC_* contracts intact. Queries in `apps/api/src/lib/queries/` return already converted to camelCase — upstream consumers don't notice the change.

**Note on `cuid() → uuid`:** IDs remain `text`, but shape differs visually (cuids vs UUIDs). Zero functional impact; flag if any code depends on ID format.

### 1.8 — Seed

`prisma/seed.ts` currently loads AgentPrompts from `agents/agent-{1-4}-*.md`. Port:

1. `scripts/generate-seed.ts` reads `agents/*.md`, extracts YAML + markdown body, emits `supabase/seed.sql` with `insert into agent_prompts (...) values (...)`.
2. Triggered by `npm run db:seed` (dev) and manually in staging/prod after initial migration.
3. Supabase seed runs automatically on local `db reset`.

```
│  next.config.ts rewrites: /api/:path* → ${API_URL}/api/:path*       │
│  + injects X-Internal-Key header via middleware                     │
└─────────────┬───────────────────────────────────────────────────────┘
              │ (server-side proxy; browser sees same-origin)
              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ apps/api (Next.js, port 3001)                                       │
│  middleware.ts: validates X-Internal-Key → 401 if missing           │
│  route handler app/api/projects/route.ts: POST                      │
│   1. parse body with Zod schema from @brighttale/shared/schemas     │
│   2. call queries.projects.create(input) — lib/queries/projects.ts  │
│   3. return { data } or { error } in standardized envelope          │
└─────────────┬───────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ queries/projects.ts                                                 │
│  1. mapToDb(input) — camelCase → snake_case                         │
│  2. supabase.from('projects').insert(row).select().single()         │
│  3. mapFromDb(row) — snake_case → camelCase                         │
│  4. returns typed Project (@brighttale/shared/types/agents)         │
└─────────────┬───────────────────────────────────────────────────────┘
              │ @supabase/supabase-js client with service_role
              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ Supabase (Postgres + RLS)                                           │
│  service_role bypasses RLS; insert accepted; row returns            │
└─────────────────────────────────────────────────────────────────────┘
```

**Flow invariants:**

- The **browser never** talks to `apps/api` directly. Only via `apps/app` rewrite. Enforced by (a) required `X-Internal-Key`, (b) restrictive CORS on `apps/api` (allowlist = `apps/app` origin only).
- The **service_role key never leaves `apps/api`**. `apps/app` has no access.
- **BC_\* contracts** (`@brighttale/shared/types/agents`) remain camelCase — mappers isolate the snake_case database.
- **All input validation happens in `apps/api`** via Zod schema from `@brighttale/shared/schemas`. `apps/app` may do optional client-side validation with the same schema, but it's not the source of truth.

### 2.2 — `apps/app/src/lib/api-client.ts`

Single typed wrapper. Every UI call goes through it:

```ts
// apps/app/src/lib/api-client.ts
import 'server-only';

const INTERNAL_KEY = process.env.INTERNAL_API_KEY!;

type ApiResult<T> = { data: T; error: null } | { data: null; error: ApiError };

type ApiError = {
  code: string;
  message: string;
  fieldErrors?: Record<string, string[]>;
};

async function request<T>(
  path: string,
  init: RequestInit = {}
): Promise<ApiResult<T>> {
  const res = await fetch(path, {
    ...init,
    headers: {
      'Content-Type': 'application/json',
      'X-Internal-Key': INTERNAL_KEY,
      ...init.headers,
    },
  });
  const json = await res.json();
  if (!res.ok) return { data: null, error: json.error };
  return { data: json.data, error: null };
}

export const api = {
  projects: {
    list: () => request<Project[]>('/api/projects'),
    create: (input: CreateProjectInput) =>
      request<Project>('/api/projects', { method: 'POST', body: JSON.stringify(input) }),
  },
};
```

**Decision:** `api-client.ts` runs **only in server components / route handlers / server actions of `apps/app`** — never in client components. Client components call server actions that call `api-client`. This guarantees `INTERNAL_API_KEY` **never** leaks into the browser bundle. Enforced by `import 'server-only'` at the top — build breaks if imported from a client component.

### 2.3 — `apps/api/src/app/middleware.ts`

```ts
import { NextResponse, NextRequest } from 'next/server';

export function middleware(req: NextRequest) {
  if (!req.nextUrl.pathname.startsWith('/api/')) return NextResponse.next();

  const key = req.headers.get('x-internal-key');
  if (key !== process.env.INTERNAL_API_KEY) {
    return NextResponse.json(
      { data: null, error: { code: 'UNAUTHORIZED', message: 'Missing or invalid internal key' } },
      { status: 401 }
    );
  }
  return NextResponse.next();
}

export const config = {
  matcher: ['/api/:path*'],
};
```

**CORS:** `apps/api/next.config.ts` sets headers allowlist for `app.brighttale.io` (prod) and `http://localhost:3000` (dev) on `/api/*` routes. Any other origin is blocked by preflight.

### 2.4 — Standardized response shape

**Success:**

```json
{ "data": <T>, "error": null }
```

**Error:**

```json
{
  "data": null,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid input",
    "fieldErrors": { "title": ["Required"] }
  }
}
```

Helper in `apps/api/src/lib/api/response.ts`:

```ts
export const ok = <T>(data: T, init?: ResponseInit) =>
  NextResponse.json({ data, error: null }, init);

export const fail = (code: ErrorCode, message: string, extras?: object, status = 400) =>
  NextResponse.json({ data: null, error: { code, message, ...extras } }, { status });
```

Closed error codes: `VALIDATION_ERROR | NOT_FOUND | CONFLICT | UNAUTHORIZED | FORBIDDEN | INTERNAL | UPSTREAM_ERROR`.

**Supabase errors** are translated by a central mapper:

- `PGRST116` (not found) → `NOT_FOUND`
- `23505` (unique violation) → `CONFLICT`
- `23503` (FK violation) → `VALIDATION_ERROR`
- RLS/permission → `FORBIDDEN`
- Rest → `INTERNAL` (full server-side log, generic message to client)

### 2.5 — Query layer: Prisma → Supabase migration pattern

**Before (`src/lib/queries/projects.ts`):**

```ts
export async function listProjects() {
  return prisma.project.findMany({
    include: { stages: true },
    orderBy: { createdAt: 'desc' },
  });
}
```

**After (`apps/api/src/lib/queries/projects.ts`):**

```ts
import { createServiceClient } from '@/lib/supabase';
import { mapProjectFromDb } from '@brighttale/shared/mappers/db';
import type { Project } from '@brighttale/shared/types/agents';

export async function listProjects(): Promise<Project[]> {
  const sb = createServiceClient();
  const { data, error } = await sb
    .from('projects')
    .select('*, stages(*)')
    .order('created_at', { ascending: false });

  if (error) throw new SupabaseError(error);
  return data.map(mapProjectFromDb);
}
```

**Universal pattern:**

1. Queries live in `apps/api/src/lib/queries/<table>.ts`.
2. Each function takes camelCase input → converts to snake_case via mapper → calls `supabase.from(...)` → converts response back to camelCase → returns the type from `@brighttale/shared/types/agents`.
3. `supabase-js` errors bubble up as `SupabaseError` — a small internal class defined in `apps/api/src/lib/api/errors.ts` that wraps the original Supabase error. Route handler translates via the code mapper from §2.4.
4. **One query per function** — no dynamic string querying. Refactors stay easy.

### 2.6 — snake_case ↔ camelCase mappers (`packages/shared/src/mappers/db.ts`)

```ts
import type { Database } from '../types/database';
import type { Project } from '../types/agents';

type ProjectRow = Database['public']['Tables']['projects']['Row'];

export function mapProjectFromDb(row: ProjectRow & { stages?: StageRow[] }): Project {
  return {
    id: row.id,
    title: row.title,
    description: row.description ?? undefined,
    status: row.status,
    createdAt: row.created_at,
    updatedAt: row.updated_at,
    stages: row.stages?.map(mapStageFromDb) ?? [],
  };
}

export function mapProjectToDb(
  input: Partial<Project>
): Partial<Database['public']['Tables']['projects']['Insert']> {
  return {
    ...(input.title !== undefined && { title: input.title }),
    ...(input.description !== undefined && { description: input.description ?? null }),
    ...(input.status !== undefined && { status: input.status }),
  };
}
```

**Rules:**

- Never pass the entire object to the DB — always construct the `Insert`/`Update` explicitly. Prevents derived or obsolete fields from leaking into storage.
- `null` vs `undefined`: DB uses `null`, TS uses `undefined`. Mapper normalizes at the boundary.
- Unit tests in `packages/shared/src/mappers/__tests__/db.test.ts` cover each entity.

### 2.8 — Seed generator in detail (`scripts/generate-seed.ts`)

**Input:** `agents/agent-{1,2,3,4}-*.md` files at repo root.

**Each file has shape:**

```markdown
---
id: agent-1-brainstorm
name: Brainstorm Agent
version: 1.0.0
---

# System Prompt

<prompt content...>
```

**Algorithm:**

```ts
import { readdirSync, readFileSync, writeFileSync } from 'node:fs';
import { join } from 'node:path';
import matter from 'gray-matter';

const AGENTS_DIR = join(process.cwd(), 'agents');
const OUT = join(process.cwd(), 'supabase', 'seed.sql');

const files = readdirSync(AGENTS_DIR).filter(f => /^agent-\d+.*\.md$/.test(f));

const inserts = files.map(f => {
  const raw = readFileSync(join(AGENTS_DIR, f), 'utf8');
  const { data, content } = matter(raw);
  const id = sqlEscape(data.id);
  const name = sqlEscape(data.name);
  const version = sqlEscape(data.version);
  const body = sqlEscape(content.trim());
  return `insert into agent_prompts (id, name, version, body, created_at, updated_at)
values ('${id}', '${name}', '${version}', '${body}', now(), now())
on conflict (id) do update set name = excluded.name, version = excluded.version, body = excluded.body, updated_at = now();`;
});

const header = `-- GENERATED by scripts/generate-seed.ts — DO NOT EDIT BY HAND\n-- Source: agents/*.md\n\n`;
writeFileSync(OUT, header + inserts.join('\n\n') + '\n');
```

**Safety:**

- `sqlEscape()` uses `$tag$...$tag$` dollar-quoting (handles quotes and newlines without escape).
- Tag chosen: `$bt$` (checks body for absence; if present, uses `$bt1$`, `$bt2$`, etc).
- Script is idempotent (`on conflict ... do update`) — safe to run multiple times.
- `seed.sql` is **committed** (future CI can verify sync by running the script and checking diff).

**When it runs:**

- Manual: `npm run db:seed` (regen + reset + apply local).
- Local dev: `supabase db reset` applies `seed.sql` automatically.
- Staging/prod: applied explicitly via `scripts/db-push-prod.sh` as a separate step after migrations (with confirmation).

### 2.9 — RLS on day 1 (even without auth)

Even without users, RLS is **enabled on all tables**. Pattern in the initial migration:

```sql
-- For each table:
alter table projects enable row level security;
-- Zero policies = deny-all for anon and authenticated roles.
-- service_role always bypasses RLS by Supabase design.
```

**Why?** Defense in depth. If someone accidentally exposes the `anon` key in `apps/app` (inevitable with Next), tables remain opaque. When real auth lands, we add policies without changing structure.

| Zod 4 in project + Zod 3 dependency from future `@tn-figueiredo/*` | High | Medium | Documented; decided when integrating auth |
| Accidental Supabase dev/prod switch | Low | Critical | `db-which.sh` always before push; `db-push-prod.sh` asks for confirmation |
| `database.ts` stale after new migration | High | Low | Manual flow: always `db:types` after alter |

### 3.6 — Acceptance criteria (definition of done)
