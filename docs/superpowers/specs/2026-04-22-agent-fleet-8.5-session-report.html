01 — O Cenário

Oito agentes.
Uma pipeline. Problemas invisíveis.

O Plan A e o Plan B da frota tinham saído. Prompts calibrados, output em português, estrutura em forma. Mas havia algo que só aparece quando você para e olha para as fronteiras — os contratos entre os componentes. O que o agente promete entregar versus o que o código espera receber.

A pipeline do Bright Tale é um organismo vivo: Brainstorm → Research → Production → Review → Publish. Cada estágio consome o output do anterior. Quando um contrato mente — quando um tipo diz que existe um campo que não existe, ou deixa de declarar um que existe — o erro não grita. Ele sussurra, em produção, semanas depois.

⚠️ A auditoria de escrutínio profundo identificou 3 Blockers, 4 Críticos e 1 NIT — todos silenciosos, todos capazes de degradar conteúdo ou crashar a pipeline em produção.
02 — O Diagnóstico

Três blockers que
ninguém via vindo.

Cada um operava em silêncio. Nenhum lançava exceção. Todos corroíam qualidade.

🔴 Blocker 1 O mapeador de keywords: conectado na teoria, desligado na prática

O agente de Research passou a emitir secondary keywords com fonte rastreável: {keyword, source_id}. Mas o agente de Production esperava lista simples. Existia uma função legacyKeywordFallback() que resolvia exatamente isso — ela converte o novo formato para o antigo. O problema: ela nunca foi chamada. O mapeador mapResearchToProductionInput ignorava a função. Em produção, keywords chegavam no formato errado e o conteúdo saía sem SEO adequado. Sem crash, sem log, sem alertas.

❌ Antes
// mapResearchToProductionInput
seo: research.seo  // passa o objeto bruto
// Production recebe: {keyword, source_id}[]
// Espera: string[]  → silently broken
✅ Depois
const normalizedSeo = research.seo ? {
  secondary_keywords:
    legacyKeywordFallback(
      research.seo.secondary_keywords
    ),
  ...rest
} : undefined;
🔴 Blocker 2 content_warning: da emissão ao vácuo

Os agentes foram instruídos a emitir content_warning quando o conteúdo fosse sensível — política, religião, saúde. O campo aparecia no YAML de output. E aí sumia. Dois gargalos o eliminavam em sequência: (1) Os schemas Zod de blog, video, shorts, podcast e engagement não declaravam o campo — Zod o cortava na validação. (2) A rota de brainstorm não persistia o campo no banco. O banner de alerta que deveria aparecer pro usuário nunca aparecia. Nunca.

❌ Antes — Schema Zod (blog)
// Zod stripeia campos não declarados
export const blogOutputSchema = z.object({
  title: z.string(),
  body: z.string(),
  // content_warning? INEXISTENTE
  // → campo sumia na validação
})
✅ Depois — 5 schemas corrigidos
export const blogOutputSchema = z.object({
  title: z.string(),
  body: z.string(),
  content_warning: z.string().optional(),
  // ✓ preservado na validação
})
🔴 Blocker 3 Score gate: uma porta que podia travar com null

O PipelineOrchestrator tem uma lógica de loop de review: score < 40 = rejeita. O Plan A substituiu score: number por um enum quality_tier. A API ainda tentava ler score diretamente — e salvava null quando não havia score numérico. O gate de rejeição simplesmente não funcionava para conteúdo novo. Conteúdo ruim podia passar despercebido.

❌ Antes
// content-drafts.ts
const reviewScore = formatReview?.score ?? null;
// Se quality_tier='reject' mas sem score numérico
// → reviewScore = null
// → gate < 40 nunca dispara
✅ Depois — deriveTier + síntese
const tier = deriveTier(formatReview);
const legacyMap = {
  excellent: 95, good: 82,
  needs_revision: 60, reject: 20
};
const reviewScore =
  formatReview?.score ?? legacyMap[tier];

Os Quatro Críticos

🟠 Crítico 1 types/agents.ts — o arquivo que mentia

O arquivo central de contratos TypeScript não refletia o que os agentes realmente emitiam. Faltavam campos que existiam nos prompts havia semanas. Tipos mentirosos são piores que ausência de tipos: passam na compilação e quebram em runtime. Solução: cirurgia precisa em 8 tipos — content_warning em todos os outputs, ResearchSecondaryKeyword como union type, research_focus_applied e depth_applied no ResearchOutput.

🟠 Crítico 2 ContentReview.score ainda era obrigatório

Com o Plan A, score passou a ser opcional no Zod. Mas o tipo TypeScript ContentReview ainda declarava score: number (obrigatório). ReviewForm recebia erros de tipo. CompletedStageSummary usava um cast feio (r as { qualityTier?: string }) para contornar um campo que já deveria estar no tipo. Ambos corrigidos.

🟠 Crítico 3 ReviewResult sem qualityTier

A interface que passa o resultado do review para o pipeline não tinha o campo qualityTier. O ReviewEngine calculava o tier internamente mas não conseguia incluí-lo no resultado que passava adiante. O pipeline recebia o resultado truncado. Corrigido: campo adicionado, tier propagado.

⚫ Crítico 4 schemas/agents.ts — código morto diagnosticado

Suspeito de estar desatualizado, o arquivo foi investigado. Resultado: era importado apenas por providerAdapter.ts, que está depreciado (todos os métodos lançam throw new Error). Nenhuma rota real o usava. Decisão: documentar, não tocar. Os schemas vivos estão em apps/api/src/lib/modules/*/schema.ts.

⚪ NIT Texto envergonhado na UI — "Score: 0/100"

Na página de drafts, quando o review usava o novo sistema de tier sem score numérico, o usuário via "Score: 0/100". Substituído por um badge de veredicto limpo: "Approved", "Needs Revision", etc.
