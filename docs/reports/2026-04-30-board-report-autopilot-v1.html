01 · Introdução

Como o autopilot saiu do desenho para uma versão funcional em dev.

O ciclo começou com um plano de três ondas: estruturar a hidratação dos engines a partir da configuração do autopilot (Foundation), ligar os gates do pipeline (preview, publish, assets) ao novo schema (Gates) e polir templates de configuração para reuso (Templates).

As três ondas foram entregues como planejado. O que veio depois foi diferente: uma fase de hardening, em que cada teste manual em dev exposte um bug real — autopilot que travava no publish, modais que vazavam para overview, dashboard que quebrava em projetos antigos, listagem de templates retornando erro.

Cada bug virou um commit isolado, com teste de regressão e push imediato. O ciclo fecha com o pipeline rodando ponta a ponta para um caso real: brainstorm escolhido pela IA, draft gerado com persona, review aprovando em uma iteração e publish disparando automaticamente para WordPress como rascunho.

Estágio atual: implementado em dev na branch feat/pipeline-autopilot-wizard-impl. Pronto para QA full-flow antes de subir para main.

02 · Desafios no caminho

Os problemas estratégicos que motivaram o ciclo.

Quatro desafios principais e quatro secundários moldaram o trabalho. Os principais bloqueavam o autopilot v1; os secundários eram dívidas que iam piorar conforme o sistema escalasse.

Principais

Autopilot incompleto

Crítico

Cinco dos sete estágios do pipeline não tinham disparo automático. O usuário tinha que abrir cada engine, clicar Run, esperar e clicar Avançar — anulando a promessa de "configurar uma vez e deixar rodando".

Dashboard de orquestração desatualizado

Alta

UUIDs expostos no topo, lista vertical sem hierarquia, sem progresso ao vivo, sem persistência do log de atividades — apresentação não condizia com a complexidade do que estava acontecendo por baixo.

Wizard sobrecarregado

Alta

Nove seções abertas simultaneamente em uma coluna única, modo como rádio sem contexto, sem template picker visual, CTA primário enterrado no fim. Setup virava obstáculo, não onboarding.

Modais e toasts vazando para overview

Alta

No modo overview, engines rodam ocultos via display:none, mas seus modais e toasts são renderizados em portal — vazando notificações sobre a interface limpa que o usuário esperava.

Secundários

Schema do pipeline_state_json obsoleto

Secundário

Sem slots para preview/publish, assets enum antigo. Projetos novos quebravam ao tentar persistir os campos novos do wizard.

Listagem de templates falhando

Secundário

Filtro PostgREST mal-construído (referencedTable aplicado em tabela base) retornava erro na primeira chamada — usuário criava template e ele nunca aparecia no dropdown.

ReviewEngine dispatchando valores stale

Secundário

O engine recarregava o draft após review, mas dispatchava o valor do cache em ref — pipeline avançava com score zero apesar do banco já ter score 82.

brainstorm_sessions.project_id nulo

Secundário

Engine não enviava projectId no body da chamada — sessões geradas dentro de um projeto ficavam órfãs no banco.

04 · Desafios ao longo do desenvolvimento

Obstáculos operacionais que apareceram durante a execução.

Distintos dos desafios estratégicos da seção 02 — estes são problemas de implementação que surgiram em tempo real e foram resolvidos antes de virarem regressão.

Test pollution entre cenários

vi.clearAllMocks() não reseta mockReturnValue, então mocks definidos em um teste vazavam para o próximo. Sintoma: o mesmo teste passava sozinho e falhava na suíte. Tratado: re-stub explícito no início de cada teste afetado.

SSE acoplado à visibilidade do modal

Suprimir o modal em overview também matou o stream SSE — pipeline travava esperando um onComplete que nunca vinha. Resolvido: effectiveUrl passou a depender de sseUrl (não de open); UI segue gated, listener segue ativo.

Subagent estourando contexto

Uma das subagents tentou modificar cinco engines simultaneamente e bateu o limite de tokens antes do commit. Tratado: dividir em ondas menores; quando estourou, conduzir manualmente o commit e seguir.

Hidratação de projetos legacy

Projetos abertos com pipeline_state_json antigo não tinham as novas slots — render quebrava. Tratado: defensive guards em StagePanelDetail e merge com defaults em handleLoadTemplate.

Recovery de projeto travado

Projeto 45ed165d tinha o brainstorm completo no banco mas a sessão e o front nunca trocaram telegramas. Resolvido: script one-shot patcheou pipeline_state_json com a ideia escolhida pela IA e avançou para research; script deletado depois.

Falhas pré-existentes contaminando sinal

Oito testes de API falham desde antes deste ciclo (research enqueue, content-drafts produce, personas WP avatar). Tratado: stash test confirmou pré-existência; falhas separadas do diff atual e flagged para próximo ciclo.

Inngest brainstorm não popula idea_archives

Sessão é criada e marcada como completed, mas o passo de persistência das ideias falha em silêncio. Anotado: não impacta autopilot v1, mas precisa investigação no próximo ciclo.

Templates legacy quebrando o wizard

Templates criados antes das slots preview/publish faziam methods.reset com config parcial — render crashava em cfg.preview.enabled. Tratado: handleLoadTemplate faz merge sobre baseConfig, getStageSummary tem guards defensivos.

06 · Conclusão

O que dá para dizer ao fim do ciclo.

Autopilot v1 funcional em dev — três ondas estruturais e oito correções pós-QA fecharam o ciclo.

O pipeline executa ponta a ponta. Brainstorm escolhe ideia, research aprova cards, draft compõe com persona, review aprova ou itera, assets e preview rodam ou pulam conforme configuração, publish vai pro WordPress como rascunho. Em supervised o usuário pausa quando quiser; em overview tudo roda silencioso com log ao vivo.

O que ainda não está fechado é uma fila de itens menores — testes do wizard pendentes de update para a nova UI, falhas pré-existentes da API, um bug separado do Inngest no brainstorm. Nada bloqueia QA.
