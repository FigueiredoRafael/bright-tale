03 · Arquitetura XStateXState architecture

Por que uma máquina de estados, e não useState.

Why a state machine, not useState.

Antes do refator, cada engine carregava seu próprio "running", "phase", "sessionId" em useState local. Funcionava — até o componente desmontar. Esta seção explica por que adotamos XState e por que evoluímos para substates compostos.

Before the refactor, each engine carried its own "running", "phase", "sessionId" in local useState. It worked — until the component unmounted. This section explains why we adopted XState and why we evolved to compound substates.

Princípio: Principle: tudo que precisa sobreviver à remontagem do componente OU informar uma decisão do pipeline pertence à máquina. O componente é projeção. anything that must survive component remount OR inform a pipeline decision belongs in the machine. The component is a projection.

Detalhe técnico · Por que XState e não Redux ou Zustand? Tech detail · Why XState and not Redux or Zustand?

Redux/Zustand são stores de dados — guardam um valor mutável. São ótimos para "estado da aplicação" (carrinho, user, tema), mas péssimos para fluxos com regras tipo "só pode ir de X para Y se Z". O desenvolvedor escreve essas regras à mão, em todo callback, e cada nova feature multiplica os caminhos de divergência.

XState é uma máquina de estados — codifica AS REGRAS. Você declara: "do estado research.generating, o evento RESEARCH_COMPLETE leva para draft, salvando o resultado". Eventos enviados de fora que não fazem sentido naquele estado são descartados silenciosamente. Isso significa: testar a máquina é testar o pipeline inteiro.

Bonus: o XState Inspector mostra o estado atual em tempo real durante o desenvolvimento — você literalmente vê a máquina caminhar pelo gráfico enquanto o usuário clica.

Redux/Zustand are data stores — they hold a mutable value. Great for "app state" (cart, user, theme), bad for flows with rules like "can only go from X to Y if Z". The developer writes those rules by hand in every callback, and each new feature multiplies divergence paths.

XState is a state machine — it encodes THE RULES. You declare: "from state research.generating, the event RESEARCH_COMPLETE moves to draft, saving the result". Events sent from outside that don't make sense in that state are silently discarded. Meaning: testing the machine is testing the entire pipeline.

Bonus: the XState Inspector shows current state in real time during development — you literally watch the machine walk the graph as the user clicks.

Detalhe técnico · Como o engine virou projeção Tech detail · How the engine became a projection

Antes: const [running, setRunning] = useState(false) — o engine sabia se estava rodando.

Depois: const isGenerating = useSelector(actor, s => s.matches({ research: 'generating' })) — o engine pergunta para a máquina.

Quando o componente desmonta (porque o usuário navegou para o overview e voltou), a máquina mantém o substate generating. Ao remontar, o engine lê do stageStatus e reconecta o stream SSE para a sessão ativa.

Before: const [running, setRunning] = useState(false) — the engine knew if it was running.

After: const isGenerating = useSelector(actor, s => s.matches({ research: 'generating' })) — the engine asks the machine.

When the component unmounts (because the user navigated to overview and back), the machine retains the generating substate. On remount, the engine reads from stageStatus and reconnects the SSE stream to the active session.

Detalhe técnico · O que entra em cada engine na fase pré-publicação Tech detail · What gets added to each engine in the pre-publish phase

Draft engine — instruções adicionadas ao prompt: títulos com menos de 70 caracteres; pelo menos 3 incidências de H3; persona escreve com objetivo EEAT (Experience: 1ª pessoa + datas + contexto temporal; Expertise: profundidade técnica + fontes citadas; Authoritativeness: schema markup + fontes confiáveis).

Review engine — três camadas:

  • Avaliador SEO pré-review: word count 1000–1200, H3 ≥ 3, presença de external links.
  • Tool de Internal Links: durante o loop de revisão, o agente pode invocar a tool que consulta a REST API do WordPress conectado ao canal e retorna posts relevantes por tag/keyword. O agente decide quais incluir.
  • Avaliador EEAT: bio do autor, profundidade técnica, fontes citadas, schema markup.

Publish engine — checagens objetivas: todas as imagens têm alt? Tem focus keyword? Tem meta description? Bloqueia publish se algum estiver faltando.

Draft engine — instructions added to the prompt: titles under 70 characters; at least 3 H3 incidences; persona writes with EEAT objective (Experience: first person + dates + temporal context; Expertise: technical depth + cited sources; Authoritativeness: schema markup + reliable sources).

Review engine — three layers:

  • SEO pre-review evaluator: word count 1000–1200, H3 ≥ 3, presence of external links.
  • Internal Links tool: during the review loop, the agent can invoke the tool that queries the channel-connected WordPress REST API and returns posts matched by tag/keyword. The agent decides which to include.
  • EEAT evaluator: author bio, technical depth, cited sources, schema markup.
