Admin Redesign — Implementation Approaches

Three strategies for achieving the visual + UX overhaul

A

Component-Level Overhaul (Recommended)

Strategy: Redesign shared primitives first (KpiCard, AdminShell sidebar, tables, charts), then restyle each page using those new building blocks.

Theme: Build a proper theme system with CSS custom properties for light/dark. Replace hardcoded Tailwind colors with semantic tokens (--surface, --card, --accent-glow).

Charts: Recharts (already React-native, tree-shakeable, composable). Sparklines as a small reusable component.

Pipeline: Custom React component with SVG connections, draggable nodes (react-dnd or native drag), animated flow lines via CSS.

Pros

  • Consistent design system across all pages
  • Theme toggle works everywhere automatically
  • Reusable — new admin pages inherit the look
  • Incremental — can ship page by page

Cons

  • More upfront work on primitives
  • Need to refactor existing page code to use new components
B

Page-by-Page Restyle

Strategy: Take each page in priority order (Agents → Dashboard → Analytics → Users → Orgs) and restyle inline. No shared component extraction — each page gets bespoke styling.

Theme: next-themes for toggle, update Tailwind classes per page with dark: variants.

Charts: Same Recharts approach.

Pipeline: Same custom component approach.

Pros

  • Faster to start — jump straight into priority pages
  • Less abstraction overhead
  • Can ship one page at a time

Cons

  • Design drift between pages
  • Duplicate styling logic
  • Theme toggle harder to maintain (scattered dark: classes)
  • New pages don't inherit the look
C

Drop External Package + Full Rebuild

Strategy: Remove @tn-figueiredo/admin dependency entirely. Rebuild AdminShell, KpiCard, and all shared components in-house using shadcn/ui primitives. Full control over every pixel.

Theme: shadcn/ui's built-in theme system (CSS variables in globals.css) — already has light/dark support.

Charts: Recharts with custom themed wrappers.

Pipeline: Same custom component approach.

Pros

  • Zero external UI dependency — full control
  • Consistent with apps/app (same shadcn foundation)
  • Theme system built into framework
  • Most maintainable long-term

Cons

  • Largest scope — need to rebuild AdminShell sidebar from scratch
  • More work upfront than A or B
