This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Structure

```
bright-tale/                    ← monorepo root (npm workspaces)
├── apps/
│   ├── app/                    ← @brighttale/app  — Next.js UI (port 3000)
│   ├── api/                    ← @brighttale/api  — Next.js API only (port 3001)
│   └── web/                    ← @brighttale/web  — landing page (port 3002)
├── packages/
│   └── shared/                 ← @brighttale/shared — types, schemas, mappers
├── supabase/
│   ├── migrations/             ← SQL migrations (apply with db:push:dev)
│   └── seed.sql                ← generated by scripts/generate-seed.ts
└── scripts/                    ← helper bash/ts scripts
```

## Development Commands (run from repo root)

```bash
# Start both app + api in parallel
npm run dev

# Or individually
npm run dev:app       # http://localhost:3000
npm run dev:api       # http://localhost:3001
npm run dev:web       # http://localhost:3002

# Build
npm run build         # all workspaces
npm run build:api

# Tests
npm run test          # all workspaces
npm run test:api
npm run test:app

# Typecheck + lint
npm run typecheck
npm run lint

# Database (requires SUPABASE_ACCESS_TOKEN in root .env.local for remote)
npm run db:push:dev   # link to dev project + push migrations
npm run db:push:prod  # link to prod project + push (requires confirmation)
npm run db:types      # regenerate packages/shared/src/types/database.ts from remote
npm run db:start      # start local Supabase (Docker required)
npm run db:stop
npm run db:reset      # reset local DB + re-run migrations + seed

# Seed agent prompts
npm run db:seed       # runs scripts/generate-seed.ts → seed.sql → supabase db reset
```

## Architecture

### Apps

- **`apps/app`** — the main UI. All `/api/*` calls are rewritten (Next.js rewrites) to `apps/api` at runtime. A Next.js middleware at `src/middleware.ts` intercepts `/api/*`, strips any client-supplied `x-internal-key` and `x-user-id` (anti-spoofing), injects the real `X-Internal-Key` from `process.env`, and adds a `x-request-id` for end-to-end tracing before the rewrite proxies the request.
- **`apps/api`** — Route Handlers only (no pages). Guarded by middleware at `src/middleware.ts` that validates `X-Internal-Key` on every `/api/*` request. Uses Supabase `service_role` key — bypasses RLS.
- **`apps/web`** — static landing page stub.

### Shared Package

`packages/shared` is consumed at source level (no dist build) via `transpilePackages` in `next.config.ts` and tsconfig `paths` aliases. Contains:

- `src/types/agents.ts` — BC_* agent contract interfaces
- `src/types/database.ts` — auto-generated Supabase DB types (run `db:types`)
- `src/schemas/` — Zod schemas for all API endpoints
- `src/mappers/db.ts` — snake_case ↔ camelCase mappers (fromDb/toDb)

### Tech Stack

- **Next.js 16** (App Router) + React 19 + TypeScript
- **Supabase JS v2** — `service_role` client in `apps/api/src/lib/supabase/index.ts`
- **Zod** for request/response validation
- **shadcn/ui** + Tailwind CSS 4 for UI
- **Vitest 4** for tests

### 4-Agent Content Workflow

5-stage pipeline: **Brainstorm → Research → Production → Review → Publish**

- Agent definitions: `agents/agent-{1-4}-*.md`
- TypeScript contracts: `packages/shared/src/types/agents.ts`
- Stage data chains via mappers: `mapBrainstormToResearchInput`, etc.
- Workflow is Claude/ChatGPT-assisted: platform generates BC_*_INPUT YAML, user pastes BC_*_OUTPUT YAML back.

### Database

Supabase (PostgreSQL). 18 tables, RLS enabled on all (deny-all — only `service_role` can read/write). `moddatetime` extension + `handle_updated_at()` trigger for `updated_at` columns.

Key tables: `projects`, `stages`, `research_archives`, `blog_drafts`, `idea_archives`, `templates` (self-referencing), `agent_prompts`, `wordpress_configs`, `ai_provider_configs`.

After schema changes: edit `supabase/migrations/`, run `npm run db:push:dev`, then `npm run db:types`.

### API Response Envelope

All API responses use `{ data, error }` shape:
- Success: `{ data: T, error: null }`
- Error: `{ data: null, error: { code, message } }`

Helpers in `apps/api/src/lib/api/response.ts`: `ok()`, `fail()`.

### Security

- `INTERNAL_API_KEY` shared secret between `apps/app` and `apps/api` — never exposed to browser. Injected by `apps/app/src/middleware.ts` on every `/api/*` request; validated by `apps/api/src/middleware/authenticate.ts`.
- `apps/app` middleware also **strips** any browser-supplied `x-internal-key` and `x-user-id` before setting its own, preventing header-spoofing attacks even if the secret leaks.
- In production, set both `API_URL` (e.g. `https://api.brighttale.io`) and `INTERNAL_API_KEY` on the Vercel project for `apps/app` — without them, Next.js rewrites target `localhost:3001` and Vercel blocks the request with `DNS_HOSTNAME_RESOLVED_PRIVATE` (404).
- `SUPABASE_SERVICE_ROLE_KEY` — `apps/api` only, never in `apps/app`.
- `NEXT_PUBLIC_SUPABASE_*` — anon key only, safe to expose (for future client-side auth).

### Path Aliases

- `@/*` → `./src/*` in each app
- `@brighttale/shared` → `../../packages/shared/src` (source, not dist)

### Testing

Vitest with jsdom. Tests in `__tests__/` directories. Category A/B tests pass without DB; Category C tests (DB-hitting) are skipped with `describe.skip` + `// TODO-test`.

Run single test: `npx vitest run apps/api/src/lib/__tests__/crypto.test.ts`

### Environment Files (gitignored)

```
.env.local                     ← root: SUPABASE_ACCESS_TOKEN for CLI
apps/app/.env.local            ← API_URL, INTERNAL_API_KEY, NEXT_PUBLIC_SUPABASE_*
apps/api/.env.local            ← INTERNAL_API_KEY, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY
```

### Canonical Decisions

- API envelope: `{ data, error }` — always, no exceptions
- Templates API: `GET /api/templates/:id` (raw) vs `GET /api/templates/:id/resolved` (merged)
- Bulk export: JSON only (ZIP deferred)
- Agent contracts: BC_* YAML format is the standard
- Legacy idea format: 30-day backward compatibility (`isLegacyIdea()`, `normalizeLegacyIdea()`)
- `shared` package: source-level consumption only — no build step
